ADR-008 Rev 2: Section 2 — Lifecycle State Machine & Transitions

How bundles move between states, what triggers transitions, and what happens on failure

State Definitions
A

ACTIVE

WAL open and appending. Memtable hot (dual rotation). Locator in RAM. File descriptors: 1 (WAL) + memtable overhead. Can read and write.

P

PARKED

WAL closed, flushed to segment. Memtable empty. Locator in RAM (for fast reads). File descriptors: 0. Can read; write transitions to Active.

C

COLD

WAL closed, flushed to segment. Memtable empty. Locator NOT in RAM (loaded lazily on first read). File descriptors: 0. Read triggers Cold→Active rebuild.

U

UNHEALTHY

Bundle failed to transition (WAL corruption, disk error, manifest failure). stateErr records the error. retryAfter controls backoff. Read/write return errors to caller.

Transition Triggers
Transition Trigger Action
NotFound → Cold First access (read or write) for tenant Check disk: if segments exist → Cold (no locator). If nothing on disk → Cold (empty).
Cold → Active WriteTraces/ReadTraces called Open WAL (new or append), rebuild locator from .trace.index files, create memtable, replay WAL.
Parked → Active WriteTraces called Reopen WAL in append mode, memtable stays empty (data already in segment). Fast.
Active → Parked time.Since(lastWrite) > ParkTimeout (default 30s) Flush memtable → segment → fsync → write checkpoint → close WAL. Keep locator in RAM.
Parked → Cold time.Since(lastRead) > ColdTimeout (default 5min) OR LRU eviction Drop locator from RAM. Bundle stays in sync.Map (for fast existence check).
Any → Unhealthy WAL open fails, manifest append fails, flush fsync fails Record stateErr, set retryAfter=now+5s. Return error to caller.
Unhealthy → Active Lifecycle monitor: now > retryAfter Attempt full rebuild (same as Cold→Active). If fails → double retryAfter (5→10→30→60s max).
Critical: No Deadlock Guarantee

Lock Hierarchy (enforced)

  • sync.Map ops: no explicit lock (Go runtime handles)
  • bundle.mu: ONLY for bundle mutable fields
  • Rule: NEVER hold bundle.mu while calling sync.Map.Store/Load
  • Rule: eviction removes from sync.Map FIRST, then locks bundle.mu

What would deadlock (and we avoid)

  • ❌ Holding bundle.mu → calling sync.Map.Delete
  • ✅ sync.Map.Delete → then lock bundle.mu → safe
  • ❌ Holding sync.Map lock (if we used RWMutex) → waiting for bundle.mu
  • ✅ sync.Map has no user-visible lock → safe
